## Why no automated tests

The fix adds DOM scroll behavior (`container.scrollTop = container.scrollHeight`) gated by a "near bottom" check (`scrollHeight - scrollTop - clientHeight < 50`). This is a pure DOM interaction:

- **jsdom limitation**: jsdom does not implement real layout. `scrollHeight`, `scrollTop`, and `clientHeight` are all 0 in jsdom, making the "near bottom" check untestable without mocking those properties.
- **Mock value**: Mocking `scrollHeight`/`scrollTop`/`clientHeight` on a container div and asserting `scrollTop` was set would test our mock setup, not the actual behavior.
- **Lexical complexity**: The `cursorAtEnd` logic is deeply coupled to Lexical's internal `$getRoot()`, `$getSelection()` etc., which require a full Lexical test environment. The existing `cursorAtEnd` logic is already untested for the same reason.

## Acceptance criteria that require manual/E2E testing

1. **During active transcription, the viewport follows the end of the transcribed text** — verify by running transcription and watching the editor scroll down as text fills the visible area.
2. **User can see the most recently transcribed content without manually scrolling** — same as above, for both long final tokens and ghost text.
3. **Does not interfere with manual scroll/cursor** — verify by:
   - Moving cursor to middle of document → start transcription → viewport should NOT scroll
   - Scrolling up with scrollbar (cursor stays at end) → new tokens arrive → viewport should NOT snap back down
   - Staying at the bottom of the document → new tokens arrive → viewport SHOULD scroll to follow
