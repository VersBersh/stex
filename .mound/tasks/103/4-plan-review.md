**Verdict** — `Needs Revision`

**Plan Issues**
1. Major — **Step 4 / Step 5** only renames `audio_final_proc_ms`, but the existing token-lifecycle semantics are wrong in a deeper way. The client does **not** use `final_audio_proc_ms` to separate final from non-final tokens; it uses `token.is_final`, then uses `final_audio_proc_ms` only as a watermark to dedupe re-sent final tokens across messages. That means the current rule “final tokens appear once and are never re-sent” is false and would remain false after this plan. Evidence: [src/main/soniox.ts](C:/code/draftable/stex/.mound/worktrees/worker-3-ba331ea9/src/main/soniox.ts):139 and [src/main/soniox.ts](C:/code/draftable/stex/.mound/worktrees/worker-3-ba331ea9/src/main/soniox.ts):149. Fix: rewrite the token-lifecycle rules and pseudocode so they say `is_final` determines token class, prior final tokens may be re-sent, and `final_audio_proc_ms` is the dedupe watermark.

2. Major — **Step 7** overstates hide-path finalization. On stop/hide, the app only sends the empty frame and waits if the socket is connected **and** there are pending non-final tokens; otherwise it skips directly to `SESSION_STOP`, optional clipboard copy, and hide. Evidence: [src/main/session.ts](C:/code/draftable/stex/.mound/worktrees/worker-3-ba331ea9/src/main/session.ts):120 and [src/main/session.ts](C:/code/draftable/stex/.mound/worktrees/worker-3-ba331ea9/src/main/session.ts):129. Fix: document hide finalization as conditional, and distinguish it from pause, which always calls `finalizeSoniox()` while recording.

3. Major — **Step 6** is incomplete for disconnect classification. The plan only names close codes `1000`, `1001`, `1006`, `1008`, and `1011`, but the implementation also has a fallback path for all other/application-defined codes and still refines those by reason text. Evidence: [src/main/error-classification.ts](C:/code/draftable/stex/.mound/worktrees/worker-3-ba331ea9/src/main/error-classification.ts):48 and [src/main/error-classification.ts](C:/code/draftable/stex/.mound/worktrees/worker-3-ba331ea9/src/main/error-classification.ts):67. Fix: include the fallback behavior, not just the named codes, and keep JSON server-error responses separate from disconnect classification.

4. Minor — **Step 2** treats `"stt-rt-v4"` as though it is the protocol’s fixed model value. In code, the client sends `settings.sonioxModel`; `stt-rt-v4` is only the current default in settings. Evidence: [src/main/soniox.ts](C:/code/draftable/stex/.mound/worktrees/worker-3-ba331ea9/src/main/soniox.ts):66 and [src/main/settings.ts](C:/code/draftable/stex/.mound/worktrees/worker-3-ba331ea9/src/main/settings.ts):43.
