**Verdict** — `Needs Fixes`

**Issues**
1. **Major — Hidden Coupling / Temporal Coupling:** The new spec documents `finished` as an independent top-level response field, but the processing algorithm only handles it after `if (!response.tokens) return;`, which means a `finished`-only response would be ignored. See [spec/api.md:80](/C:/code/draftable/stex/.mound/worktrees/worker-3-ba331ea9/spec/api.md#L80), [spec/api.md:97](/C:/code/draftable/stex/.mound/worktrees/worker-3-ba331ea9/spec/api.md#L97), [spec/api.md:146](/C:/code/draftable/stex/.mound/worktrees/worker-3-ba331ea9/spec/api.md#L146), [src/main/soniox.ts:135](/C:/code/draftable/stex/.mound/worktrees/worker-3-ba331ea9/src/main/soniox.ts#L135), and [src/main/session.ts:19](/C:/code/draftable/stex/.mound/worktrees/worker-3-ba331ea9/src/main/session.ts#L19). This matters because finalization in the session layer waits on `finished`, so the protocol now relies on an unstated assumption that `finished` always arrives with `tokens`. That is a semantic coupling the spec should remove or make explicit. Suggested fix: either document `finished` responses as always including `tokens: []`, or change the processing pseudocode/spec wording so `finished` is handled independently of `tokens`.

2. **Minor — SRP / Abstraction Leakage:** The “Client-Side Processing” section bakes renderer-specific behavior into the protocol spec by saying clients should “Commit new final tokens to the Lexical editor as EditorBlocks.” See [spec/api.md:130](/C:/code/draftable/stex/.mound/worktrees/worker-3-ba331ea9/spec/api.md#L130), [spec/api.md:161](/C:/code/draftable/stex/.mound/worktrees/worker-3-ba331ea9/spec/api.md#L161), [src/main/soniox.ts:25](/C:/code/draftable/stex/.mound/worktrees/worker-3-ba331ea9/src/main/soniox.ts#L25), and [src/main/soniox-lifecycle.ts:9](/C:/code/draftable/stex/.mound/worktrees/worker-3-ba331ea9/src/main/soniox-lifecycle.ts#L9). The implementation is actually layered: the Soniox client emits token events and does not depend on Lexical or `EditorBlock`. Mixing those concerns makes the architecture less legible and makes the API doc harder to extend for other consumers. Suggested fix: describe this step in neutral terms such as “commit finalized tokens to the client’s transcript state/UI,” and leave Lexical-specific mapping to a renderer or feature doc.
