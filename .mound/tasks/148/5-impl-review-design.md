1. **Major — Hidden coupling / side effects: `push()` retains caller-owned mutable buffers by reference**. In [src/main/audio-ring-buffer.ts:21](C:/code/draftable/stex/.mound/worktrees/worker-2-9f2bd432/src/main/audio-ring-buffer.ts#L21), `push()` stores `chunk` directly (`this.chunks.push({ data: chunk, startMs })`). That makes correctness depend on every caller treating the buffer as immutable after handoff. Nothing in the API enforces or communicates that ownership rule, so a caller that reuses or mutates a scratch `Buffer` can silently corrupt the ring buffer’s retained history and any replay produced by `sliceFrom()`. This is a real semantic/data coupling problem for a persistence-oriented component. Fix by copying on write (`const stored = Buffer.from(chunk)`) before storing, or by making ownership explicit in the API contract and tests. The current tests in [src/main/audio-ring-buffer.test.ts](C:/code/draftable/stex/.mound/worktrees/worker-2-9f2bd432/src/main/audio-ring-buffer.test.ts) do not cover this contract.
