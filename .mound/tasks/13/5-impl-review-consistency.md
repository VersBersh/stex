**Issues**
1. Major: The undo/redo guarantee is not actually preserved by only mutating `historyState.current`. In [TokenCommitPlugin.tsx](/C:/code/draftable/stex/.mound/worktrees/worker-12-bf191077/src/renderer/overlay/editor/TokenCommitPlugin.tsx#L27), final-token updates are tagged `'historic'`, and then only `historyState.current` is replaced at [TokenCommitPlugin.tsx](/C:/code/draftable/stex/.mound/worktrees/worker-12-bf191077/src/renderer/overlay/editor/TokenCommitPlugin.tsx#L44). Lexical’s history listener explicitly returns early for discarded historic updates before it clears `redoStack` or rewrites prior snapshots in [LexicalHistory.dev.mjs](/C:/code/draftable/stex/.mound/worktrees/worker-12-bf191077/node_modules/@lexical/history/LexicalHistory.dev.mjs#L244), while undo/redo later restore whole old editor snapshots from `undoStack`/`redoStack` in [LexicalHistory.dev.mjs](/C:/code/draftable/stex/.mound/worktrees/worker-12-bf191077/node_modules/@lexical/history/LexicalHistory.dev.mjs#L170). That means an existing redo entry, or a deeper undo chain created before transcription arrives, can still restore a pre-transcription state and remove committed text, which violates [spec/models.md](/C:/code/draftable/stex/.mound/worktrees/worker-12-bf191077/spec/models.md#L57). Fix by updating or invalidating the existing undo/redo stacks when final tokens are committed, rather than updating `current` alone.

2. Major: The new `EditorBlock[]` can diverge from the actual document as soon as the user clears the editor. The manager is created privately inside [Editor.tsx](/C:/code/draftable/stex/.mound/worktrees/worker-12-bf191077/src/renderer/overlay/editor/Editor.tsx#L31) and passed only to the plugin at [Editor.tsx](/C:/code/draftable/stex/.mound/worktrees/worker-12-bf191077/src/renderer/overlay/editor/Editor.tsx#L44). Existing clear behavior only wipes Lexical in [OverlayContext.tsx](/C:/code/draftable/stex/.mound/worktrees/worker-12-bf191077/src/renderer/overlay/OverlayContext.tsx#L32), but never calls `blockManager.clear()`. Because [editorBlockManager.ts](/C:/code/draftable/stex/.mound/worktrees/worker-12-bf191077/src/renderer/overlay/editor/editorBlockManager.ts#L21) merges new Soniox text into the last unmodified Soniox block, the next final tokens after a clear can attach to stale internal blocks while the visible editor starts from empty. That breaks the requirement that blocks represent the full document in [spec/models.md](/C:/code/draftable/stex/.mound/worktrees/worker-12-bf191077/spec/models.md#L35). Fix by hoisting the block manager into shared overlay/session state and wiring `clear()` into every editor-reset path.
