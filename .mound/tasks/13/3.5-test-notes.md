# Test Notes — T13: Token Commit & EditorBlock Management

## Test Files Created

### `src/renderer/overlay/editor/editorBlockManager.test.ts`

Pure unit tests for the `EditorBlockManager` module:

- **commitFinalTokens — basic**: Creates new soniox block from first batch
- **commitFinalTokens — extension**: Extends existing unmodified soniox block with consecutive batches
- **commitFinalTokens — multi-token**: Concatenates multiple tokens within a single batch
- **commitFinalTokens — empty tokens**: No-ops for empty arrays and empty text
- **commitFinalTokens — unique IDs**: Each new block gets a unique ID
- **block boundaries — modified block**: Does not extend a modified soniox block; creates a new one
- **block boundaries — after user block**: Creates a new soniox block after a user-source block
- **getDocumentText**: Returns concatenated text from all blocks
- **clear**: Removes all blocks; allows new blocks after clear
- **independent managers**: Separate instances have independent state

## Acceptance Criteria Not Automatically Tested

- **Programmatic appends excluded from undo history**: This requires a Lexical editor with DOM environment (JSDOM/happy-dom). The `'historic'` tag + `historyState.current` approach was verified by reading the Lexical HistoryPlugin source code (`@lexical/history/LexicalHistory.dev.mjs`). Integration testing of undo behavior would require a full React/Lexical test harness which is beyond the scope of this task.
- **Final tokens appended to editor as permanent text**: This involves the `TokenCommitPlugin` Lexical component. Testing requires a DOM and React rendering context. The plugin logic is straightforward (append TextNode to last paragraph), and the block manager tests verify the data model side.
- **Ghost text removal on final token arrival**: Deferred to the ghost text feature task.
