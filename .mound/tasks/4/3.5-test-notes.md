# Test Notes for T4: Window Manager

## Test Files Created

- `src/main/window.test.ts`

## What Each Test Covers

### `initWindowManager`
- Creates overlay window hidden (`show: false`)
- Creates overlay as frameless (`frame: false`)
- Creates overlay as always-on-top
- Creates overlay with skip-taskbar
- Creates overlay with correct default size (600x300)
- Creates overlay with minimum size constraints (400x200)
- Creates overlay with saved size from settings
- Loads overlay renderer HTML
- Registers `before-quit` handler on app

### `getOverlayWindow`
- Returns `null` before init
- Returns the window instance after init

### `showOverlay`
- Shows the overlay window
- Restores saved position when valid

### `hideOverlay`
- Hides the overlay window
- Saves position on hide
- Saves size on hide
- Is a no-op if window is not visible

### `toggleOverlay`
- Shows overlay when hidden
- Hides overlay when visible

### `position validation`
- Resets position when saved position is off all displays
- Keeps valid position on a connected display
- Validates across multiple displays

### `opacity on focus/blur`
- Sets opacity to 1.0 on focus
- Sets opacity to 0.95 on blur

### `close interception`
- Converts close to hide when app is not quitting
- Allows close when app is quitting

### `showSettings`
- Creates a settings window
- Creates settings as a standard framed window
- Creates settings with normal taskbar behavior
- Loads settings renderer HTML
- Focuses existing settings window if already open

## Acceptance Criteria Coverage

| Criterion | Test(s) |
|-----------|---------|
| Overlay is frameless, always-on-top, no taskbar | `initWindowManager` tests |
| Position/size persistence across show/hide | `hideOverlay` saves, `showOverlay` restores |
| Position validation on monitor disconnect | `position validation` tests |
| Opacity changes on focus/blur | `opacity on focus/blur` tests |
| Settings window as separate standard window | `showSettings` tests |
| Both windows load respective HTML | `initWindowManager` + `showSettings` load tests |

## Untested

- **Fade-in animation timing**: The 100ms fade-in uses `setTimeout` internally and is difficult to test precisely in unit tests without timer mocking. The implementation will use `vi.useFakeTimers()` if feasible, but the visual smoothness is ultimately a manual verification.
- **Debounced move/resize persistence**: The debounce timing is an implementation detail; we test that `setSetting` is called with position/size on hide (which is the reliable save point).
