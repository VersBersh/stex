**Plan Issues**
1. Critical ‚Äî Step 6: the proposed `AUDIO_CAPTURE_ERROR` flow leaks renderer-side capture resources. In the plan, main clears `capturing` before invoking the stored error callback; then [`onAudioError()`](/C:/code/draftable/stex/.mound/worktrees/worker-1-300188a2/src/main/soniox-lifecycle.ts#L137) calls `stopCapture()`, but the planned `stopCapture()` would now be a no-op once `capturing` is already false, so no `audio:stop-capture` command reaches the overlay. That leaves the `MediaStream` / `AudioContext` / worklet running after an error. Fix: make the error path send `audio:stop-capture` before state is cleared, or centralize cleanup in `stopCapture()` and let the error handler call that first.

2. Major ‚Äî Steps 6-7: the plan regresses device selection on fresh installs by replacing the current preload/API flow with raw `navigator.mediaDevices.enumerateDevices()` in the settings renderer. Today the settings UI intentionally gets devices through [`window.settingsApi.getAudioDevices()`](/C:/code/draftable/stex/.mound/worktrees/worker-1-300188a2/src/renderer/settings/index.tsx#L33), backed by the typed bridge in [`preload.d.ts`](/C:/code/draftable/stex/.mound/worktrees/worker-1-300188a2/src/shared/preload.d.ts#L37) and [`settings-preload.ts`](/C:/code/draftable/stex/.mound/worktrees/worker-1-300188a2/src/preload/settings-preload.ts#L6). With Chromium enumeration, labels are commonly empty until mic permission has already been granted, and [`window.ts`](/C:/code/draftable/stex/.mound/worktrees/worker-1-300188a2/src/main/window.ts#L255) has no permission/bootstrap handling for the settings window. Fix: keep the `settingsApi.getAudioDevices()` contract and reimplement it on the browser side behind the preload, with an explicit permission strategy, instead of bypassing the bridge and returning `[]` from main.

3. Major ‚Äî Step 10: test coverage is incomplete for the API contract the plan changes. If `getAudioDevices()` is no longer meant to be used, the plan also needs to update or remove [`settings-preload.ts`](/C:/code/draftable/stex/.mound/worktrees/worker-1-300188a2/src/preload/settings-preload.ts#L20), [`preload.d.ts`](/C:/code/draftable/stex/.mound/worktrees/worker-1-300188a2/src/shared/preload.d.ts#L42), and [`settings-preload.test.ts`](/C:/code/draftable/stex/.mound/worktrees/worker-1-300188a2/src/preload/settings-preload.test.ts#L65). As written, the plan leaves a typed public API in place, makes main return an empty fallback, and does not update the preload tests. Fix: either preserve the API and test it, or remove it cleanly everywhere.

4. Major ‚Äî Step 6: the plan does not explicitly update audio-error normalization/classification even though the implementation source changes from PortAudio errors to browser `getUserMedia`/DOMException errors. The current classifier in [`error-classification.ts`](/C:/code/draftable/stex/.mound/worktrees/worker-1-300188a2/src/main/error-classification.ts#L3) only keys off message substrings, while the plan forwards only a plain message string from renderer. That is brittle for `NotAllowedError`, `NotFoundError`, `NotReadableError`, and similar browser failures. Fix: send structured error data (`name`, `message`) or normalize errors in the renderer before IPC, and add an explicit step to update classifier/tests.

5. Minor ‚Äî Step 8: dependency removal is incomplete. The repo has a [`package-lock.json`](/C:/code/draftable/stex/.mound/worktrees/worker-1-300188a2/package-lock.json), but the plan only mentions [`package.json`](/C:/code/draftable/stex/.mound/worktrees/worker-1-300188a2/package.json) and webpack cleanup. Fix: include lockfile regeneration/update in the dependency-removal step.
