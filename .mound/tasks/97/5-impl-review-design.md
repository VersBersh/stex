**Issues**
1. **Major — Hidden coupling / Primitive obsession in audio device identity**. The new flow exposes, stores, and resolves devices purely by display name: [src/main/audio.ts#L33](C:/code/draftable/stex/.mound/worktrees/worker-2-19bfef77/src/main/audio.ts#L33), [src/main/audio.ts#L87](C:/code/draftable/stex/.mound/worktrees/worker-2-19bfef77/src/main/audio.ts#L87), [src/shared/preload.d.ts#L37](C:/code/draftable/stex/.mound/worktrees/worker-2-19bfef77/src/shared/preload.d.ts#L37), and [src/renderer/settings/pages/General.tsx#L68](C:/code/draftable/stex/.mound/worktrees/worker-2-19bfef77/src/renderer/settings/pages/General.tsx#L68). That only works if `AudioDevice.name` is unique and stable, which is not enforced anywhere and is not a safe assumption for USB headsets, virtual devices, or duplicated system labels. `startCapture()` then takes the first matching name, so the user can silently get the wrong microphone. This is a real semantic-coupling problem between renderer selection and main-process resolution. Fix by introducing a small device DTO for IPC with separate identity and label, and persist a deliberate selector instead of the raw display string. If numeric IDs are not stable enough across sessions, use a richer token/fallback strategy in main rather than collapsing the model to `string`.

2. **Minor — The refresh behavior introduces temporal coupling that is only implicit in the UI controller**. In [src/renderer/settings/index.tsx#L33](C:/code/draftable/stex/.mound/worktrees/worker-2-19bfef77/src/renderer/settings/index.tsx#L33) and [src/renderer/settings/index.tsx#L41](C:/code/draftable/stex/.mound/worktrees/worker-2-19bfef77/src/renderer/settings/index.tsx#L41), device freshness now depends on a window focus transition. That is pragmatic, but it means the list stays stale while the settings window remains focused, and the refresh policy is buried in the root component rather than communicated by the device-selection boundary itself. This is a mild SRP / embedded-design issue, not a structural failure. A better shape would be to factor the behavior into a named `refreshAudioDevices` path or a dedicated hook/service so the policy is explicit, and add a manual refresh affordance if focus-only refresh is the intended tradeoff.
