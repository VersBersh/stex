  1. **Major — Hidden Coupling / Temporal Coupling:** [`src/main/theme.ts#L26`](//C:/code/draftable/stex/.mound/worktrees/worker-5-d8d05577/src/main/theme.ts#L26), [`src/main/index.ts#L9`](//C:/code/draftable/stex/.mound/worktrees/worker-5-d8d05577/src/main/index.ts#L9), [`src/main/theme.test.ts#L42`](//C:/code/draftable/stex/.mound/worktrees/worker-5-d8d05577/src/main/theme.test.ts#L42). `initThemeManager()` registers a global IPC handler plus two long-lived listeners every time it is called, but the module does not enforce single initialization or offer disposal. The current app startup happens to call it once, so production is probably fine today, but the design now depends on that call order remaining true forever. In real Electron, re-registering `ipcMain.handle` for the same channel throws; repeated init would also duplicate settings/native-theme subscriptions. The test double masks that by silently overwriting handlers, so the tests currently reinforce the hidden assumption instead of catching it. Suggested fix: make `initThemeManager()` idempotent with an `initialized` guard or return a disposer/reset hook, and update the `ipcMain.handle` mock to match Electron’s duplicate-registration behavior.
  2. **Minor — Interface Integrity / Dependency on a Looser Contract:** [`src/main/preload.ts#L8`](//C:/code/draftable/stex/.mound/worktrees/worker-5-d8d05577/src/main/preload.ts#L8), [`src/main/preload-settings.ts#L21`](//C:/code/draftable/stex/.mound/worktrees/worker-5-d8d05577/src/main/preload-settings.ts#L21), [`src/renderer/types.d.ts#L3`](//C:/code/draftable/stex/.mound/worktrees/worker-5-d8d05577/src/renderer/types.d.ts#L3), [`src/renderer/settings/settingsApi.d.ts#L3`](//C:/code/draftable/stex/.mound/worktrees/worker-5-d8d05577/src/renderer/settings/settingsApi.d.ts#L3). The preload implementations type the theme callback payload as plain `string`, while the public renderer contract says `"light" | "dark"`. That weakens the abstraction boundary and makes the design slightly less self-describing: the shared `ResolvedTheme` type exists, but the boundary that should enforce it does not use it. Suggested fix: import and use `ResolvedTheme` in both preload files so the runtime bridge and renderer declarations express the same contract.
