**Issues**
1. **Major — Hidden coupling / Dependency boundary: the IPC and persistence boundary trusts unvalidated external data.** [settings.ts](/C:/code/draftable/stex/.mound/worktrees/worker-7-bd725a82/src/main/settings.ts#L36) [settings.ts](/C:/code/draftable/stex/.mound/worktrees/worker-7-bd725a82/src/main/settings.ts#L51) [types.ts](/C:/code/draftable/stex/.mound/worktrees/worker-7-bd725a82/src/shared/types.ts#L40)  
   `getSettings()` casts `store.store` to `AppSettings`, and `SETTINGS_SET` casts `unknown` to `AppSettings[typeof key]` before persisting it. That gives compile-time comfort but no runtime safety. A malformed IPC payload or a corrupted on-disk store can produce an object that is typed as `AppSettings` while violating the domain contract (`theme`, `windowSize`, etc.). This is a real design problem: high-level code now depends on unstated assumptions about renderer correctness and store contents.  
   **Fix:** add runtime validation at the boundary. A per-setting validator map or a schema for the full `AppSettings` object would be enough. Validate before `store.set`, validate/repair when reading, and add tests for invalid IPC payloads and corrupt persisted values.

2. **Major — DRY / Hidden semantic coupling: window geometry now has two sources of truth.** [settings.ts](/C:/code/draftable/stex/.mound/worktrees/worker-7-bd725a82/src/main/settings.ts#L19) [window.ts](/C:/code/draftable/stex/.mound/worktrees/worker-7-bd725a82/src/main/window.ts#L6)  
   The new defaults define `windowSize: { width: 600, height: 300 }`, while `createOverlayWindow()` separately hard-codes `width: 600` and `height: 300`. That duplication is semantic coupling: the code only stays correct while both modules happen to evolve together. If one changes and the other does not, the persisted settings model and actual window behavior diverge.  
   **Fix:** make one place authoritative. Either have `window.ts` read initial bounds from `getSettings()`, or extract a shared constant used by both the settings defaults and window creation.

3. **Minor — SRP / Temporal coupling: `registerSettingsIpc()` is a one-shot bootstrap operation, but the API and tests hide that constraint.** [settings.ts](/C:/code/draftable/stex/.mound/worktrees/worker-7-bd725a82/src/main/settings.ts#L48) [index.ts](/C:/code/draftable/stex/.mound/worktrees/worker-7-bd725a82/src/main/index.ts#L5) [settings.test.ts](/C:/code/draftable/stex/.mound/worktrees/worker-7-bd725a82/src/main/settings.test.ts#L196)  
   The method both defines the IPC contract and mutates global Electron state. That means it must be called in the right order and only once, but nothing in the function name or contract communicates that. The test double further hides this by allowing repeated `handle()` calls to overwrite existing handlers, so the tests bless a usage pattern the real Electron API typically rejects.  
   **Fix:** either keep registration in a dedicated bootstrap module and avoid exporting it as a reusable service API, or make it explicitly idempotent. Tighten the mock so duplicate registration fails the same way production would.
