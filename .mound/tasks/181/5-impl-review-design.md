That is a semantic/temporal coupling to Soniox batch cadence and network jitter, not to an explicit “utterance finished” signal. If a continuation arrives after `300ms`, the plugin will commit a partial word, clear `pendingRef`, and later treat the continuation as a new immutable final token sequence. That breaks the merger contract and there is no repair path once `commitWords()` has run. The comment itself documents this as “typical” timing, which means correctness is based on observation rather than an enforced interface. **Suggested fix:** drive the flush from an explicit upstream boundary (`<end>`, endpoint/finalization event, or another main-process abstraction), or at minimum thread a real utterance-ended signal into the renderer instead of inferring one from elapsed time.
