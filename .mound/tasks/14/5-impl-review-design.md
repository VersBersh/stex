2. Major — **Clean code / Semantic coupling**: the cursor-tracking logic is narrower than the documented contract and will misclassify some legitimate “at end” and “mid-document” selections. In [TokenCommitPlugin.tsx](C:/code/draftable/stex/.mound/worktrees/worker-11-e629c669/src/renderer/overlay/editor/TokenCommitPlugin.tsx#L47) the code decides `cursorAtEnd` solely from `prevSelection.anchor` being exactly the last text node at its text length. That ignores whether the selection is collapsed, ignores `focus`, and ignores end-of-paragraph element positions. The spec added in [inline-editing.md](C:/code/draftable/stex/.mound/worktrees/worker-11-e629c669/spec/features/inline-editing.md#L18) and [inline-editing.md](C:/code/draftable/stex/.mound/worktrees/worker-11-e629c669/spec/features/inline-editing.md#L43) requires preserving any mid-document selection and tracking only a caret at the tail. As written, a reversed selection whose anchor happens to be at the tail can be treated as “track end”, and a caret represented on the paragraph element at the end can be treated as mid-document. Fix by treating “track end” only for collapsed selections at the true document tail, and otherwise always restoring the full saved range.

3. Major — **Hidden coupling / Architecture clarity**: the new inline-edit path assumes a single-paragraph document, but that assumption is neither enforced nor modeled. [InlineEditPlugin.tsx](C:/code/draftable/stex/.mound/worktrees/worker-11-e629c669/src/renderer/overlay/editor/InlineEditPlugin.tsx#L18) diffs `$getRoot().getTextContent()`, which includes `\n` between paragraphs, while [editorBlockManager.ts](C:/code/draftable/stex/.mound/worktrees/worker-11-e629c669/src/renderer/overlay/editor/editorBlockManager.ts#L46) concatenates block text with no separators.
