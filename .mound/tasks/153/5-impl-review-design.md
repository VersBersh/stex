It is not cleared on disconnect/reconnect or when a new `SonioxClient` is created. That means stale non-final timing from connection A can survive into connection B, and `capturePendingStartMs()` can combine that stale value with the current `connectionBaseMs`. The design therefore depends on an unstated call order: “a new connection must emit non-finals before pause is allowed to snapshot state.” Nothing in the API enforces that. Fix by making pending-token tracking part of per-connection state, or at minimum clearing it whenever a connection is torn down or replaced, and add a reconnect-path test for this case.

2. **Major — Embedded Design / Primitive Obsession Around Time Domains:** [`src/main/soniox-lifecycle.ts:80`](/C:/code/draftable/stex/.mound/worktrees/worker-3-027c4723/src/main/soniox-lifecycle.ts#L80), [`src/main/soniox-lifecycle.ts:242`](/C:/code/draftable/stex/.mound/worktrees/worker-3-027c4723/src/main/soniox-lifecycle.ts#L242), [`src/main/soniox-lifecycle.ts:289`](/C:/code/draftable/stex/.mound/worktrees/worker-3-027c4723/src/main/soniox-lifecycle.ts#L289), [`src/shared/types.ts:1`](/C:/code/draftable/stex/.mound/worktrees/worker-3-027c4723/src/shared/types.ts#L1). The lifecycle now mixes two timestamp domains behind plain `number`s and the same `SonioxToken` shape: `lastNonFinalStartMs` stores raw connection-relative `start_ms`, while forwarded tokens are session-relative after `applyTimestampOffset()`. A reader cannot infer that from the code alone. `capturePendingStartMs()` only works because it remembers to reapply `connectionBaseMs` later. That is semantic coupling, not explicit design. Fix by normalizing pending-start tracking to session time immediately when non-final tokens arrive (`baseMs + tokens[0].start_ms`), or by introducing explicit raw/session-relative types so the contract is encoded instead of implied.
