**Plan Issues**
1. **Critical — Step 3 (`flush on clear`)**. The current clear lifecycle does not support the ordering the plan assumes. `clearEditor()` clears the Lexical root first, then runs registered clear hooks in [OverlayContext.tsx](C:/code/draftable/stex/.mound/worktrees/worker-1-d675197b/src/renderer/overlay/OverlayContext.tsx#L53), and `TokenCommitPlugin` already has a clear hook that immediately calls `blockManager.clear()` in [TokenCommitPlugin.tsx](C:/code/draftable/stex/.mound/worktrees/worker-1-d675197b/src/renderer/overlay/editor/TokenCommitPlugin.tsx#L29). That means a plugin-level “flush before clear” is not implementable as written, and a “flush after clear” either repopulates text the user just cleared or gets wiped by the existing block-manager reset. Fix: either remove `clear` from the flush requirements, or explicitly add a pre-clear lifecycle in `OverlayContext` and include that extra scope in the plan.

2. **Major — Steps 1 and 2 contradict each other about when a word is complete**. Step 1 says “all groups except the last are complete words” and the last group becomes `newPending`, which is the only approach consistent with leading-space boundaries. But Step 2’s example `[" hello", " world"] -> two words` treats the final group as immediately committable. That would prematurely emit the trailing word instead of buffering it until a later boundary or pause/stop flush. The observed fragmentation in [debug-seed.json](C:/code/draftable/stex/.mound/worktrees/worker-1-d675197b/src/renderer/overlay/editor/debug-seed.json#L11) also supports the need to keep the tail buffered. Fix: rewrite the tests/examples so the last group stays pending unless a later boundary or explicit flush completes it.

3. **Major — The plan adds a new commit path but only proposes pure-function tests**. `TokenCommitPlugin` does more than append text: it preserves/restores selection, scroll-follows at the tail, and clears undo/redo state on each committed append in [TokenCommitPlugin.tsx](C:/code/draftable/stex/.mound/worktrees/worker-1-d675197b/src/renderer/overlay/editor/TokenCommitPlugin.tsx#L80). A pause/stop flush is another commit path with the same history/cursor risks, but the plan only adds `tokenMerger.test.ts`. Fix: add integration coverage for pause/stop flush behavior, including block-manager/editor consistency, cursor tracking, and history reset behavior.
