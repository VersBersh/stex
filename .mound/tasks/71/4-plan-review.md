**Plan Issues**
1. Major — Step 1 / Step 3 / Step 5: the plan treats Lexical paragraph serialization as an investigation-only detail, but the rest of the plan and the proposed spec text already hard-code `\n`. The implementation depends on `$getRoot().getTextContent()` in [InlineEditPlugin.tsx](C:/code/draftable/stex/.mound/worktrees/worker-12-24cdfe01/src/renderer/overlay/editor/InlineEditPlugin.tsx#L23), and there is no existing test that locks down how multiple paragraphs are flattened. Pure `findTextDiff` and block-manager tests can all pass even if Lexical emits a different separator or handles a trailing empty paragraph differently. Fix: make Step 1 a committed regression test against real Lexical paragraph nodes, then derive the Step 3 expectations and spec wording from that result.

Major — Step 6: the plan explicitly leaves the `InlineEditPlugin` / `UserTypingPlugin` interaction unverified even though both listeners run on every non-`historic` update from the same editor instance in [Editor.tsx](C:/code/draftable/stex/.mound/worktrees/worker-12-24cdfe01/src/renderer/overlay/editor/Editor.tsx#L47), [InlineEditPlugin.tsx](C:/code/draftable/stex/.mound/worktrees/worker-12-24cdfe01/src/renderer/overlay/editor/InlineEditPlugin.tsx#L23), and [UserTypingPlugin.tsx](C:/code/draftable/stex/.mound/worktrees/worker-12-24cdfe01/src/renderer/overlay/editor/UserTypingPlugin.tsx#L13). Fix: add at least one integration-style editor test, or make manual verification of Enter/split plus subsequent typing an explicit deliverable.

3. Minor — Step 4: the “token commit after paragraph break” case is underspecified and its rationale is partly wrong. The plan says “new soniox block is created (since previous block is modified)” without defining which setup it means. Fix: split this into two explicit tests, or state the exact initial blocks, edit offset, and expected final block sequence.
